# 01 — Catálogo de requisitos

> Cada requisito: **ID · Enunciado · Fuente · Justificación**. Las especificaciones (`02`–`07`) referencian estos IDs; la matriz (`08`) verifica que todos quedaron cubiertos.

## A. Requisitos funcionales (RF)

| ID | Enunciado | Fuente | Justificación |
|---|---|---|---|
| **RF-01** | El sistema permite **crear una implementación** eligiendo uno de los tres casos: `migration` (Caso 1), `new_client` (Caso 2), `special_module` (Caso 3). | [PROC] §1–3 | Los tres procesos existen y son distintos desde la entrada; el caso determina etapas, campos, scripts y correos. |
| **RF-02** | Al elegir el caso, el sistema muestra **solo** las etapas, campos y validaciones de ese caso (plantilla de proceso). | [PROC] §4 (tabla comparativa) | Las diferencias entre casos son numerosas y precisas (usuarios, BD, scripts, pruebas); mostrar campos de otro caso induce errores de captura. |
| **RF-03** | Los Casos 1 y 2 inician con la etapa de **filtro de Ventas** (¿opciones existen en SAG Web? → ¿beneficio en híbrido?) con salida posible **Rechazar** (fin del proceso). El Caso 3 **no** tiene filtro: entra directo. | [PROC] §1.A.2, §2.A.2, §3 nota inicial | El Caso 3 "por definición ya es híbrido"; imponerle el filtro contradiría el proceso. La rama de rechazo es una salida real del negocio y debe quedar trazada (quién rechazó y por qué). |
| **RF-04** | El sistema registra la **solicitud a Elasticserver** y su contenido varía por caso: C1 = dominio + BD de pruebas + acceso a BD de producción **por compañía**; C2 = dominio + BD nueva; C3 = dominio siempre, BD de pruebas solo si el módulo la requiere, ninguna BD si el cliente es local. | [VEN] §D.1, [CTX] §3.1, [DEC] A | Es el primer correo de la cadena y sus variantes son reglas de negocio firmes (decisiones resueltas). |
| **RF-05** | El sistema genera y envía el correo de **prerrequisitos al cliente**, con variante por caso, y para el Caso 3 **dos variantes**: BD local y nube. | [VEN] §D.2 | Es el correo "mejor presentado" (HTML) del proceso; sus reglas de contenido difieren por caso y sub-caso. |
| **RF-06** | El sistema captura los **entregables estructurados** del caso (espejo del `Checklist de Ventas - {caso}`) y calcula su **completitud** automáticamente. | [VEN] §B | El bucle "¿entregables completos?" del backbone se decide hoy a ojo; estructurar los campos permite calcularlo y bloquear el avance con faltantes. |
| **RF-07** | El sistema registra el evento **"solicitar faltantes"** (a cliente o entre áreas) sin retroceder la etapa, contabilizando cuántas idas y vueltas hubo. | [PROC] §0 (bucles del backbone) | Los bucles de faltantes existen en los tres flujos; medirlos es valor nuevo (dónde se atascan las implementaciones) sin distorsionar el estado. |
| **RF-08** | La **entrega a Soporte** es una acción única que: valida completitud, envía **un solo correo** con toda la información **indicando el caso**, y transfiere la responsabilidad a Soporte. | [VEN] §D.3, regla C.5 | "Un solo correo con todo + el caso" es regla explícita del contrato de documentación. La transferencia de dueño habilita la bandeja por responsable. |
| **RF-09** | El **paso a paso técnico** de cada caso existe en el sistema como **checklist instanciado por implementación**, con fases (A–F), orden fail-fast, responsable por paso, estado (`pending/in_progress/done/blocked/not_applicable`) y campo de **evidencia**. | [PROC] §1.B, §2.B, §3.B, §5 | El paso a paso es el corazón operativo. En Fase 1 no se ejecuta nada automáticamente: se registra QUÉ se hizo, QUIÉN y CON QUÉ resultado (evidencia), que es la trazabilidad pedida. |
| **RF-10** | Un paso marcado **`blocked` en FASE A detiene el avance** de la etapa técnica y muestra el motivo. Desbloquear exige registrar cómo se resolvió. | [PROC] §5 (filosofía fail-fast), [SQL] §D | El reorden fail-fast existe precisamente para detenerse temprano (correos duplicados/vacíos → STOP hasta que el cliente corrija). El software debe reforzar esa disciplina, no solo documentarla. |
| **RF-11** | Las **decisiones** del flujo (¿opciones existen?, ¿híbrido?, ¿requiere pruebas primero? (C3), ¿pruebas exitosas?) se registran con valor, autor, fecha y nota. | [PROC] §1.A, §2.A, §3.A.9 | Son las bifurcaciones del proceso; sin registro no hay trazabilidad de por qué una implementación tomó una rama. |
| **RF-12** | Para el Caso 3, la decisión **"¿requiere ambiente de pruebas?"** se apoya en un **catálogo de módulos** (WMS → pruebas; portales, Power BI → directo a producción), editable por admin. | [PROC] §3.A.9, [DEC] B.4 | La clasificación existe pero está incompleta ("lista por completar"); un catálogo editable evita recompilar cuando Ventas/Soporte la completen. |
| **RF-13** | El sistema envía el correo de **entrega de credenciales** distinguiendo **pruebas** y **producción**, con URL de acceso, portal de fallos (Zoho) y la guía de logo/formato de impresión referenciada. | [VEN] §D.4 | Es el correo de cierre de cada ambiente; sus elementos (Zoho, guía de logo — sin formato asignado SAG Web falla al imprimir) son requisitos explícitos de la plantilla actual. |
| **RF-14** | Toda acción relevante genera un **evento inmutable** en la línea de tiempo de la implementación (creación, cambio de etapa, dato editado, paso completado, decisión, correo enviado/fallido, faltantes pedidos, escalamiento, cierre). | Pedido de Juan Camilo (trazabilidad); [PROC] §0 | Es el requisito central de la Fase 1: "adding traceability of how the process is going on". |
| **RF-15** | Existe la acción **"Escalar a Ingeniería"**: registra el evento y envía notificación por correo; Ingeniería **no** es un rol del flujo. | [CTX] glosario (Ingeniería), [DEC] A | El proceso es explícito: "Ingeniería NO es actor del flujo; solo recibe escalamientos". Modelarla como rol le daría permisos que el proceso niega. |
| **RF-16** | Una **bandeja/tablero** lista las implementaciones agrupadas por etapa, con filtros (caso, cliente, responsable) y bandera **"Requiere atención"** (paso bloqueado o sin movimiento > N días). | Pedido de Juan Camilo; [SYS] patrón de Actualizaciones programadas | Reutiliza el patrón ya validado en el sistema (agrupación por estado + bandera de atención) que el usuario ya aprobó para programaciones. |
| **RF-17** | Al **cerrar** una implementación (`completed`), el sistema ofrece **sembrar los maestros**: crear Cliente, Dominio y Bases de datos en el sistema existente y sugerir la primera Actualización programada. *(Fase 3; el gancho queda especificado desde ya.)* | [SYS]; diseño §8 | Evita recapturar datos que este mismo sistema administra; conecta la implementación con el ciclo de vida de actualizaciones que es la razón de ser del sistema. |
| **RF-18** | Los correos del flujo se envían **desde el sistema** con **vista previa obligatoria** (asunto + HTML renderizado) y quedan registrados (destinatario, fecha, resultado) en la historia. | Pedido de Juan Camilo; [VEN] §D | "As we gather all the data we can send the emails directly from it". La vista previa protege contra enviar datos incompletos a un externo (Elasticserver, cliente). |
| **RF-19** | El sistema **no permite enviar** un correo del flujo si falta un dato obligatorio de esa plantilla (los `{{placeholders}}` desaparecen: o hay dato, o no se envía). | [VEN] regla C.8 | La regla actual es "direcciones/datos reales, nunca inventados; si falta, placeholder y se pide". En el software el equivalente correcto es bloquear el envío, no enviar con huecos. |
| **RF-20** | El detalle de una implementación tiene tres vistas: **Avance** (etapas + checklist), **Datos** (entregables editables según rol×etapa) y **Historia** (línea de tiempo + correos). | Pedido de Juan Camilo | Separa las tres preguntas del usuario: ¿en qué va?, ¿qué información hay?, ¿qué pasó y quién lo hizo? |

## C. Requisitos no funcionales (RNF)

| ID | Enunciado | Fuente | Justificación |
|---|---|---|---|
| **RNF-01** | **Sin secretos en claro ni en correos:** credenciales SQL en Key Vault con el patrón existente (referencia + revelar/copiar auditado). Ningún correo del flujo lleva contraseñas SQL. | [SYS] políticas vigentes; [VEN] D.4 (placeholders) | Política de seguridad ya establecida en el sistema y consistente con la regla documental de "nunca datos reales en las plantillas". |
| **RNF-02** | **Autorización en el backend** por rol × etapa (la UI solo refleja). Toda mutación valida rol, etapa actual y transición válida. | [SYS] `permissions.ts` + JWT | La seguridad no puede depender del frontend; el sistema ya sigue este patrón. |
| **RNF-03** | **Trazabilidad inmutable:** los eventos no se editan ni borran; `metadata` pasa por la sanitización de auditoría existente (password/secret/token/jwt). | [SYS] `audit.ts` | Un registro de trazabilidad editable no sirve como evidencia; la sanitización ya existe y se reutiliza. |
| **RNF-04** | **Compatibilidad SQL Server:** entidades con FKs naturales; arrays → tablas hijas; enums → CHECK; sin features exclusivos de Cosmos. Se documenta el mapeo. | [SYS] matriz de migración existente | Restricción vigente del proyecto (migración futura a SQL ya planificada y documentada). |
| **RNF-05** | **Plantillas de proceso versionadas:** cada implementación guarda `templateVersion`; cambiar la plantilla no altera implementaciones en curso. | Buena práctica de workflow; [DEC] (los procesos aún evolucionan) | Los procesos tienen pendientes abiertos; sin versionado, cada ajuste de proceso corrompería las implementaciones abiertas. |
| **RNF-06** | **Idempotencia y concurrencia:** transiciones de etapa idempotentes (repetir la misma transición no duplica eventos ni correos); doble clic protegido. | [SYS] lección del bug de doble-delete de schedules | Bug real ya sufrido en este sistema (404 de Cosmos por doble clic); se especifica desde el diseño. |
| **RNF-07** | **Todo en español** (UI, correos, eventos legibles), consistente con el resto del sistema. | [SYS] | Los usuarios del flujo (Ventas, Soporte) trabajan en español; el sistema completo ya es en español. |
| **RNF-08** | **Correos responsivos** con el estándar existente: `layout()` (máx. 700px, tabla, paleta corporativa, preheader, CTA). | [SYS] `emailTemplates.ts` | Estándar ya aprobado por el usuario para todos los correos del sistema. |
| **RNF-09** | Los envíos usan el proveedor configurado (SMTP con contraseña en Key Vault / mock en desarrollo) y el registro de notificaciones existente. | [SYS] `emailService.ts` | Reutilización directa; no se introduce un segundo mecanismo de correo. |
| **RNF-10** | **Paginación y filtros** en la bandeja con el helper existente (`getPagination`/`paginateArray`). | [SYS] | Patrón ya presente en todas las listas del sistema. |
